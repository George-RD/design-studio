# Review: Accessibility lens

Audit a rendered web surface for WCAG accessibility gaps. Loaded as a **conditional lens** under `references/review/polish.md` when the surface is interactive or the user asked about a11y/keyboard/states. Findings feed `harness-output/review/findings.json` and `report.md`.

This is one leaf with four sections. The orchestrator may spawn a single a11y subagent that covers all four sections, or fold them into the interaction fan-out — the section split is for procedure, not a requirement to run four agents.

## When to use

Load this lens when:

- The reviewed surface is **interactive** (forms, buttons with state, modals, menus, app UI).
- The user explicitly asked about accessibility, keyboard focus, or contrast.
- `polish.md` classified the surface as `interactive` or the user named a11y.

## Not when

- The surface is **static** (marketing/landing/content with no meaningful widgets beyond links) and the user only asked for visual polish — skip this lens unless a11y was requested.
- The work is a redesign/overhaul — that is the Studio lane, not a Review lens.

## Inputs

| Input | Required | Notes |
|-------|----------|-------|
| `surface` | yes | Path and/or URL and/or running `harness-output/serve.json` from `polish.md` |
| `screenshots` | yes | Desktop 1440 + mobile 390 captures from `polish.md` |
| `surface_class` | yes | `interactive` (this lens is skipped otherwise) |
| `constraints` | no | Explicit scope ("only contrast", "skip forms") |

Grounded in live screenshots + real interaction. Never a code-only review (BOC: probe to first available browser adapter; HALT and record if none).

## Section 1: Contrast and color

Check every text and meaningful UI element against WCAG AA:

- **Normal text (under 18pt, or under 14pt bold):** contrast ratio **4.5:1** minimum against the background.
- **Large text (18pt+ or 14pt+ bold):** contrast ratio **3:1** minimum.
- **UI components and graphical objects** (borders, icons, form borders, focus rings): contrast ratio **3:1** against adjacent colors.
- **Color-only signaling is a failure.** Any meaning carried by color alone (red = error, green = success, required = asterisk) must also be conveyed by text, icon, or pattern. Flag status that disappears for color-blind users.
- Flag low-contrast placeholder text, disabled-looking text that is still readable-but-faint, and gradient backgrounds that drop text contrast below threshold in places.

## Section 2: Semantic HTML and structure

Check the document is machine-readable, not just visually arranged:

- **Heading levels** flow in order (one `h1`, no skipped levels like `h2` → `h4`). Flag heading-like styling on `div`/`span` with no semantic tag.
- **Buttons vs divs.** Interactive controls use `<button>`/`<a>`, not clickable `<div>`/`<span>` with handlers. Flag `role` fakes that a native element would solve.
- **Labels.** Every input has an associated `<label>` (or `aria-label`/`aria-labelledby`). Flag placeholder-only or floating-label-only inputs with no programmatic label.
- **Alt text.** Informative images have meaningful `alt`; decorative images have empty `alt=""`. Flag missing or keyword-stuffed alt.
- **Landmarks.** Page regions use `<header>`/`<nav>`/`<main>`/`<footer>`/`<aside>` or ARIA landmarks so assistive tech can skip and jump. Flag content outside any landmark, or duplicate/nested landmarks.
- Flag `lang` missing on `<html>`, and tables used for layout.

## Section 3: Keyboard and focus

Check the surface is fully operable without a pointer:

- **Tab reachability.** Every interactive control is reachable by Tab and activates with Enter/Space. Flag controls that trap focus or cannot be reached.
- **Tab order** follows visual/logical order. Flag jumps that confound the reading order.
- **Escape closes modals/menus/popovers**, and focus returns to the trigger. Flag overlays that trap or orphan focus.
- **Visible `:focus-visible` ring** on every focused control. Flag invisible focus or focus lost against the background.
- **No bare `outline: none`** without a replacement focus style. Flag any `outline: none` (or `outline: 0`) that removes the ring and supplies nothing.
- Flag skip-links absent on long pages and carousel/scroll regions with no keyboard pause.

## Section 4: Motion, forms, and misc

- **`prefers-reduced-motion`.** Respect it: replace meaningful motion (parallax, autoplay, long transitions) with an opacity or state fallback. Flag animations that ignore the media query.
- **Form errors.** Inline, specific, and associated with the field (`aria-describedby` / `aria-invalid`). Flag errors announced only by color, or cleared on input without user action.
- **Hit targets.** Interactive controls are at least **44x44px** where relevant (touch/responsive). Flag sub-44px tap targets crammed together.
- Flag autoplaying audio/video without pause, and timeouts that expire a session without warning or extension.

## Output

Emit one finding per issue. Each finding follows the shared Review schema so it merges into `harness-output/review/findings.json`:

```json
{
  "id": "a11y-001",
  "lens": "a11y",
  "severity": "blocker | quality | polish",
  "confidence": "high | medium | low",
  "summary": "short statement of the gap",
  "evidence": "screenshot path + element/screen + observed behavior",
  "status": "open"
}
```

- `blocker` — contrast below AA, keyboard-unreachable control, bare `outline: none`, missing label on a required field, color-only status with no text alternative.
- `quality` — skipped heading level, div-as-button, missing landmark, Escape not returning focus, form error by color only.
- `polish` — sub-44px hit target, reduced-motion fallback missing on decorative motion, alt text that could be clearer.

Include the **WCAG reference** when known (e.g. `WCAG 1.4.3`, `WCAG 2.1.1`, `WCAG 2.4.7`). Do not self-censor "minor" issues; `polish.md` merges and deduplicates, highest severity wins on overlap.
